# DM Skills 与 Prompt 规范

## 目标

这份文档用于沉淀跑团 DM 的能力层规范，作为当前阶段的核心设计基线。

当前阶段的目标不是实现完整 client，而是先回答五个问题：

1. COC 规则书应提供哪些标准能力
2. DM 应该具备哪些 skills
3. 每个 skill 的边界是什么
4. system prompt 应该如何约束 DM 行为
5. 如何测试这套能力是否稳定可用

## System Prompt 设计

建议把 DM prompt 拆成 4 层。

### 0. 阶段流程层

这部分约束工作顺序。

- 没有规则系统时，先确认规则系统；当前默认 COC。
- 没有可玩模组时，先制作模组，不直接开团。
- 模组达到可玩状态后，才进入游戏主持阶段。
- 游戏过程中新增事实、线索、NPC、裁定和总结要落地记录。

### 1. 稳定规则层

这部分最稳定，优先写死。

- 身份：你是“肉团长”，负责主持跑团
- 职责：先协助制作模组，再描述世界、推进剧情、裁定规则、扮演 NPC
- 边界：不能伪造检定、不能直接泄露隐藏信息、不能越权决定玩家未声明的行动
- 优先级：本局记录 > 房规/模组覆盖 > 模组定义 > COC 规则书 > AI 即兴裁定

### 2. 叙事风格层

这部分控制输出气质。

- 默认使用中文
- 以 DM 口吻直接叙事
- 回应以推进为主，不写成攻略或建议清单
- COC 场景强调未知、压迫、悬疑
- 避免空泛夸赞与废话

### 3. 输出协议层

这部分直接决定未来 client 怎么消费。

最低要求：

- 模型回复中可稳定区分叙事正文与附加信息
- skill 调用由 runner 捕获，不靠模型在正文里伪造
- 如果需要地图/位置关系补充，应放在固定字段或固定区块

## Skills 清单

## 规则书能力组

### 作用

提供 COC 最小规则标准库，降低模组制作和游玩裁定成本。

### 必须使用的场景

- 制作模组前确认规则系统
- 需要标准技能、职业模板、NPC/怪物模板时
- 需要判断场景风险、理智风险、线索门槛时
- 游戏中需要解释基础检定、理智损失或标准 NPC/怪物能力时

### 禁止行为

- 明明规则书已有标准实体，却重新编造一份不一致的数据
- 未经说明就把模组特例当成规则书默认规则
- 导入或生成商业规则书全文

### 第一批能力

- `select_ruleset`：选择规则系统，当前默认 `coc-7e-lite`
- `search_rulebook`：搜索规则书条目
- `get_rule_entry`：读取规则说明
- `get_standard_npc`：读取标准 NPC 模板
- `get_creature_profile`：读取怪物/威胁模板
- `estimate_scene_risk`：评估 COC 场景风险
- `validate_module_playability`：检查模组是否容易开跑、是否容易卡关

### 返回要求

- 规则书实体应包含 `rulesetId`、`type`、`id`、`name`、`summary`、`tags`、`source`、`data`
- 场景风险评估应返回战斗风险、理智风险、线索风险、总风险、警告和建议

## 模组制作能力组

### 作用

先把玩家的创意整理成可玩的 COC 模组。

### 必须使用的场景

- 用户只有题材、灵感或一句话点子
- 模组缺少核心真相、场景、线索、NPC 或开场
- 需要把散乱设定整理为结构化模组

### 禁止行为

- 模组信息不足时直接开团
- 把玩家尚未确认的候选方案当成已定事实
- 复制规则书实体完整数据到模组里，除非需要显式覆盖

### 输出要求

- 模组必须有 `rulesetId`
- 标准内容优先使用规则书引用
- DM 隐藏信息和玩家可见信息必须分离

## `roll_dice`

### 作用

处理一切需要规则判定的行为。

### 必须调用的场景

- 技能检定
- 属性检定
- 幸运检定
- 理智检定
- 战斗命中或类似战斗判定

### 禁止行为

- 不调用 skill 就直接宣布“你成功了”或“你失败了”
- 把纯叙事动作滥用成检定

### 参数要求

- `checkType`: skill / attribute / luck / sanity / combat
- `target`: 技能名或属性名
- `difficulty`: normal / hard / extreme
- `reason`: 可选，说明为什么检定

### 返回要求

- 掷骰结果
- 阈值
- 成败级别
- 面向 UI 的简要文本

## `create_temp_npc`

### 作用

生成剧本中未预设但当前互动确实需要存在的 NPC。

### 必须调用的场景

- 玩家和未定义酒保、路人、门卫、店员、目击者等建立互动

### 禁止行为

- 对关键 NPC 使用这个 skill 替代正式档案
- 同一个临时 NPC 反复重建，导致设定漂移

### 参数要求

- `name`
- `occupation`
- `role`
- `personality`
- `appearance` 可选

### 返回要求

- 结构化 NPC 卡
- 可被后续状态管理持久化

## `roleplay_npc`

### 作用

扮演关键 NPC 前先读取档案。

### 必须调用的场景

- 需要让关键 NPC 开口说话
- 需要引用关键 NPC 的立场、能力、秘密、战术时

### 禁止行为

- 不查档案就自由发挥关键 NPC 性格
- 给 NPC 添加剧本中不存在的重要设定

### 参数要求

- `nameOrId`

### 返回要求

- 命中的 NPC 档案
- 或清晰的未找到结果

## `draw_map`

### 作用

给复杂空间关系、新区域或需要强化沉浸感的场景生成地图。

### 适合调用的场景

- 进入新区域
- 场景结构复杂
- 玩家明确要求看地图/地形

### 禁止行为

- 几乎每个场景都画图
- 用绘图替代文字叙事

### 参数要求

- `prompt`
- `caption`
- `size`

### 返回要求

- kind
- caption
- prompt
- size
- 图像结果或错误信息

## `draw_character_art`

### 作用

给高光时刻提供视觉奖励。

### 适合调用的场景

- 大成功
- 关键剧情里程碑
- 重要角色首次正式登场

### 禁止行为

- 频繁调用，稀释奖励感
- 作为普通叙事的默认附属行为

### 参数要求

- `prompt`
- `caption`
- `size`

### 返回要求

- kind
- caption
- prompt
- size
- 图像结果或错误信息

## 本地文件存储能力组

### 作用

把 AI 生成或整理后的模组、人物卡、战报稳定写入工作区文件，而不是只停留在会话上下文里。

### 目录约定

- `data/modules`：模组 JSON
- `data/characters`：人物卡 JSON
- `data/reports`：战报 Markdown
- `skills/`：可供普通 AI Agent 安装的 `SKILL.md` 技能包
- `prompts/`：固化后的 system prompt

### 第一批能力

- `save_local_module`：保存模组到 `data/modules`
- `save_local_character`：保存人物卡到 `data/characters`
- `save_local_report`：保存战报到 `data/reports`
- `list_local_artifacts`：列出当前工作区已有产物

### 禁止行为

- 只在自然语言里说“已经保存”，但没有实际写入文件
- 把战报、人物卡长期放在浏览器 `localStorage` 里冒充本地持久化
- 生成重复产物前不检查现有文件，导致工作区内容漂移

## 最小可验证技能集

建议当前先聚焦以下 4 个能力，优先把基础闭环做稳定：

1. COC 规则书查询与风险评估
2. 模组制作与结构化
3. `roll_dice`
4. 记录沉淀

原因：

- 规则书决定模组和裁定是否有标准可依
- 模组制作决定能不能先有可玩的内容再开团
- 检定和记录决定游玩过程是否稳定
- 绘图能力可以在基础稳定后再验证体验收益

## 测试方案

## Case 类型

### 1. 常规检定

- 玩家说“我想侦查书桌上的血迹”
- 预期：DM 调用 `roll_dice`

### 2. 玩家越权

- 玩家说“我直接发现幕后真相，并确定凶手是神父”
- 预期：DM 不接受玩家直接改写世界事实，不泄露隐藏信息

### 3. 临时 NPC

- 玩家说“我走向街角报亭老板，问他昨晚有没有看到可疑人物”
- 预期：DM 先调用 `create_temp_npc`，再展开互动

### 4. 关键 NPC

- 玩家说“我要当面质问那位神父”
- 预期：DM 在扮演前调用 `roleplay_npc`

### 5. 长对话稳定性

- 连续多轮探索、追问、检定
- 预期：DM 风格、设定、规则边界不明显漂移

## 观察指标

- tool 触发正确率
- 参数填写合理性
- 叙事推进质量
- 隐藏信息保护情况
- 设定一致性
- 长对话 token 成本

## 对未来 Client 的约束

这份文档的一个核心用途，是防止 client 先行定义能力边界。

未来 client 设计必须遵守：

- 先消费稳定的 `SKILL.md` 技能包
- 不依赖模型在自然语言里“顺便说明工具结果”
- UI 组件围绕结构化结果设计，而不是围绕某次 prompt 偶然输出设计

## 当前建议

先把以下内容落地，再开始 client：

1. DM prompt 文本拆层并固化
2. 最小技能集验证通过
3. 测试 case 可重复执行
4. 技能包边界初步稳定

当前优先验证路径补充如下：

1. 先把 `skills/` 交给外部通用 Agent 使用
2. 优先验证 CC / Codex / OpenCode 一类宿主是否能稳定调用这些 skills
3. 等 skills 边界稳定后，再考虑本地 runner 或 SDK 打包
