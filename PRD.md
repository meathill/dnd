# 跑团助手 产品需求文档 (PRD)

## 1. 项目背景与当前判断

原本设想是基于 `@openai/agents` 构建一个纯浏览器运行的 DM Agent，让 Agent Loop、Tools 与 UI 都在前端完成闭环。

但在实际验证后，这条路线目前并不理想：

- `@openai/agents` 虽然提供了 Agent 抽象，但并没有很好解决跑团场景下的前端可控性问题。
- 跑团对状态、工具反馈、表现层联动、过程可观测性有很强要求，仅靠 SDK 默认 Agent Loop 很难做出稳定可控的浏览器体验。
- 如果过早开发 client，很容易把 UI、协议、工具定义和 prompt 设计绑死在未经验证的实现上。

因此，项目当前阶段调整为：**先沉淀跑团所需的规则书、Skills + System Prompt，并验证效果；验证通过后，再开发 client 去消费这些能力。**

## 2. 核心目标

当前阶段不追求“先把 Agent 跑起来”，而是优先建立一套可测试、可迭代、可复用的 DM 能力底座。

核心目标分为四层：

0. **引入规则书**
   - 当前优先支持 COC，先做项目自定义的 `coc-7e-lite` 最小规则包。
   - 规则书作为标准库，提供技能、职业模板、NPC/怪物模板、检定规则、理智损失与场景风险评估。
   - 模组制作时优先引用规则书实体，避免每次从零生成标准内容。

1. **定义能力**
   - 明确 DM 的 system prompt。
   - 明确各个 skills 的职责、入参、出参、触发条件、约束边界。
   - 让模型行为尽量落在结构化、可预测的范围内。

2. **验证能力**
   - 使用脚本化测试样例验证 prompt 和 skills 的组合效果。
   - 重点观察是否会乱用工具、漏用工具、越权叙事、破坏设定、输出风格漂移。
   - 在没有正式 client 之前，先获得一套足够稳定的 DM 行为规范。

3. **消费能力**
   - 等能力边界稳定后，再开发 client。
   - client 的职责是消费模型输出与工具调用结果，把它们可视化，而不是反过来定义 Agent 应该怎么工作。

## 3. 产品分阶段路线

### 3.1 第一阶段：COC 规则书能力定义

本阶段先建立 COC 最小规则书，作为模组制作和游戏主持的底层标准库。

- `coc-7e-lite` 规则集
- 技能 / 职业 / NPC 模板 / 怪物模板 / 风险评估规则
- 规则书查询与实体引用能力
- 模组中的规则书引用结构

### 3.2 第二阶段：DM 能力定义

本阶段的交付物是“协议”和“规范”，不是完整 UI。

- DM system prompt 初稿
- 跑团 skills 清单与调用约束
- 工具返回结果的数据结构约定
- 一组覆盖典型跑团场景的测试用例

### 3.3 第三阶段：DM 能力测试

围绕真实跑团流程做验证：

- 玩家自由行动
- 检定触发
- 临时 NPC 生成
- 关键 NPC 扮演
- 地图/立绘等视觉奖励型能力触发
- 长对话下的行为稳定性

这一阶段的重点不是“模型回答得像不像”，而是：

- 是否按规则调用正确 skill
- 是否遵守剧本隐藏信息边界
- 是否保持 DM 口吻与节奏
- 是否能在多轮中保持状态一致

### 3.4 第四阶段：Client 集成

在前两阶段稳定后，再开发 client，职责包括：

- 渲染文本叙事
- 渲染工具调用卡片
- 管理前端状态树
- 将用户输入、工具结果、游戏状态组织成模型可消费的上下文

client 层应该建立在已验证的 skill contract 之上，而不是与 prompt/skills 一起混合试错。

## 4. DM System Prompt 设计原则

System Prompt 需要明确 DM 的职责、边界与输出风格。

### 4.1 角色定位

- 你是“肉团长”，负责主持 COC / DND 等跑团。
- 你负责先协助制作模组，再基于模组描述世界、推进剧情、裁定规则、扮演 NPC。
- 你不是陪聊助手，也不是通用问答助手。
- 当前默认规则系统是 COC，优先使用 `coc-7e-lite` 规则书能力。

### 4.2 行为边界

- 不直接伪造检定结果，涉及判定时必须调用对应 skill。
- 不越权修改玩家角色状态，除非通过规则或工具结果得到依据。
- 不直接泄露 DM 隐藏信息，只能通过线索、事件、环境和 NPC 对话逐步揭示。
- 不把回答写成建议清单、攻略清单或系统说明文档。
- 没有可玩的模组时，先制作模组，不直接开团。
- 制作模组时，优先复用规则书实体；只有规则书缺失或模组确实需要原创时才生成新内容。

### 4.3 叙事风格

- 以 DM 口吻直接叙事。
- 保持简洁推进，避免空泛赞美和过长铺陈。
- 优先响应玩家行动带来的世界反馈。
- 在 COC 场景中保持悬疑、压迫、未知感。

### 4.4 输出结构

建议统一为结构化分段，至少包含：

- 叙事正文
- 可选的场景/位置简图
- 工具调用结果对应的附加内容

最终格式应以测试阶段的稳定性为准，再固化给 client。

## 5. Skills 规划

第一批围绕 COC 跑团核心闭环，只做最必要的 skills。

### 5.0 规则书能力

- 用途：查询 COC 规则、标准技能、职业模板、NPC/怪物模板，并评估场景风险。
- 原则：模组制作和游戏主持都优先查询规则书，不靠模型临场编标准数据。
- 返回：规则书条目、实体档案、风险评估结果或可玩性建议。

### 5.1 `roll_dice`

- 用途：处理属性、技能、理智、幸运、战斗等检定。
- 原则：所有需要判定的行动都优先通过该 skill 完成，不允许模型口头虚构掷骰。
- 返回：检定类型、目标、掷骰结果、阈值、成功级别、说明文本。

### 5.2 `create_temp_npc`

- 用途：玩家与未预设 NPC 互动时，快速生成一张临时人物卡。
- 原则：一旦生成，后续对该 NPC 的描述和扮演应保持一致。
- 返回：结构化 NPC 档案。

### 5.3 `roleplay_npc`

- 用途：在扮演关键 NPC 前，先读取其设定档案，减少设定漂移。
- 原则：关键 NPC 的说话方式、立场、能力、已知信息都要受档案约束。
- 返回：NPC 档案或未命中结果。

### 5.4 `draw_map`

- 用途：进入新区域、复杂场景辨识、空间关系强调时触发。
- 原则：这是增强体验的视觉技能，不应滥用。
- 返回：地图描述、标题、图片结果或错误信息。

### 5.5 `draw_character_art`

- 用途：作为大成功、关键剧情节点、重要角色首次登场的视觉奖励。
- 原则：稀缺使用，强调奖励感。
- 返回：立绘描述、标题、图片结果或错误信息。

## 6. 测试策略

在 client 开发前，必须先完成 prompt + skills 的验证。

### 6.1 测试目标

- 工具选择是否正确
- 工具参数是否合理
- 叙事是否符合剧本和规则
- 多轮对话中角色与世界状态是否稳定
- 极端输入下是否出现越权、跑题、泄露隐藏信息等问题

### 6.2 测试样例类型

- 基础开场：进入剧本后的第一轮互动
- 常规检定：侦查、图书馆、说服、潜行、战斗
- 异常行为：玩家试图直接改写世界或索要隐藏真相
- 临时互动：与无预设 NPC 对话
- 奖励场景：大成功或里程碑事件触发绘图
- 长流程回归：多轮连续对话后验证风格和记忆稳定性

### 6.3 测试通过标准

- 模型大多数情况下能触发正确 skill
- 输出格式和语气基本稳定
- 关键边界问题可通过 prompt 或 skill contract 收敛
- client 已经可以据此设计稳定的数据消费接口

## 7. 技术策略

当前技术策略调整为：**先做能力层，再做消费层。**

- 不把“纯浏览器 Agent Loop”作为当前里程碑。
- 不强依赖某个 Agent SDK 的默认运行范式。
- 允许先用服务端、脚本或最小运行器来测试 prompt 与 tools。
- 等能力层稳定后，再决定 client 是直接消费模型输出、消费工具调用协议，还是包一层自定义 runner。

## 8. 当前结论

这个项目的关键不是“有没有 Agent SDK”，而是：

- DM 的 system prompt 是否足够稳定
- skills 是否定义清楚
- 模型是否能在真实跑团场景中可靠使用这些 skills

所以当前正确的开发顺序是：

1. 定义 COC 规则书能力
2. 定义 skills
3. 定义 system prompt
4. 测试与迭代
5. 开发 client
