# 跑团助手 产品需求文档 (PRD)

## 1. 项目背景与痛点
原本的跑团助手采用传统的“Chatbot 线性对话”架构（配合 `input-analyzer` 和 `action-executor`）。在实际使用中发现，跑团（基于 DND / COC 规则）具有极强的状态依赖性、随机性和规则约束，单纯的聊天架构无法支撑复杂的跑团情境判定。

## 2. 核心升级目标：Agent + Skills 架构
我们将架构全面升级为 **Agent + Skills** 模式。
核心设计理念：**不需要 Multi-Agent（多智能体）**。观察人类 DM 可知，整场游戏全靠一个 DM 把控节奏。因此，我们只需要构建一个足够强大的**单 DM Agent**，并赋予它极其丰富的“技能（Skills/Tools）”来规范其行为即可。

## 3. 核心产品机制

### 3.1 纯前端智能体模式 (Browser-Side Agent)
- 跑团游戏本质上是**重客户端状态**的应用。
- 为了实现极致的响应和游戏化体验，Agent 的运行核心（Loop）和技能库（Tools）将**全部放在浏览器前端运行**。
- 后端仅负责极简的鉴权和模型请求转发代理（Proxy）。

### 3.2 具象化的工具调用 (Visualized Tool Cards)
- 传统的 Agent 只有文本输出，而在跑团助手中，**“各种工具调用都要有对应的卡片让用户有感”**。
- 例如：DM 决定使用 `roll_dice` 工具时，前端不再是干巴巴地输出数字，而是要展示真实的掷骰子微动画。
- 采用前端渲染 Agent 生命周期状态（如“思考中”、“技能释放中”），配合精美的 `<ToolCard />` 组件展示。

---

## 4. 核心技能 (Skills) 规划

随着跑团的深入，DM Agent 必须具备以下核心技能池：

1. **掷骰子判定 (`roll_dice`)**
   - 触发条件：遇到属性检定、攻击判定时。
   - 表现：触发前端 3D/CSS 骰子动画，并向 Agent 传回投掷结果。
2. **创建临时人物卡 (`create_temp_npc`)**
   - 触发条件：玩家临时决定与未准备的非关键 NPC（如路边酒保）沟通。
   - 机制：像人类 DM 一样，当场生成并“抽取”一张新的人物卡，一旦生成就固定下来存入前端状态树。
3. **扮演 NPC (`roleplay_npc`)**
   - 触发条件：与已有记录的 NPC 深度对话。
   - 机制：规范 AI DM 的扮演行为，使其符合角色卡设定。
4. **绘制地图 (`draw_map`)**
   - 触发条件：进入新区域或玩家请求辨识地形。
   - 表现：展示地图卡片。
5. **绘制立绘 (`draw_character_art`)**
   - 触发条件：一般作为玩家掷出“大成功（暴击）”或完成一段重要故事的**视觉奖励**。

## 5. 技术底座选型
经过调研 Vercel AI SDK、Claude Agent SDK、OpenAI Agents JS 与 pi-mono 等框架，最终决定采用：
- **`@openai/agents` SDK**：作为前端 Agent 的驱动核心，其支持在浏览器中运行完善的 Agent Loop 和工具链管理，并且方便通过改写 `baseURL` 的方式支持任意兼容 OpenAI API 的开源/闭源模型（如 DeepSeek）。
