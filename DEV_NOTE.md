# DEV NOTE

这里记录开发过程中积累的需要长期关注的事情、架构决策及技术反思。

## 架构演进：Browser-Side Agent (纯前端智能体) 架构思考

在跑团助手项目中，我们决定将 Agent Loop 移至纯前端（浏览器）中运行，配合 Next.js 后端作为无状态的 API Proxy。

### 采用此架构的优势：
1. **极致的 UI 联动**：工具（Tools）即前端 JS 函数。Agent 调用 `roll_dice` 可以直接无缝唤起前端 3D 骰子动画，不再需要复杂的 Server-Sent Events (SSE) 状态序列化解析。
2. **状态共享自然**：跑团游戏拥有大量 Client-Side State（地图迷雾、当前角色位置、背包数据），前端 Agent 可以以 `O(1)` 的成本直接读取内存状态，而不用每次序列化发给服务器。
3. **去中心化服务器压力**：Agent 复杂的状态循环转移到了客户端设备。

### 技术决策记录 (ADR) 与讨论路径

在确认"纯前端 Agent 架构"时，我们评估了四项重大技术挑战，并确立了极具实用主义和游戏化思维的技术路线。以下是详细的讨论路径与决策结论：

#### 1. API 代理层被"薅羊毛"的防御策略 (Proxy Security)
* **引发讨论的挑战**：如果前端负责 Agent 的运转，后端只提供单纯的 `/api/llm-proxy`，恶意用户可以拦截接口，将其当作免费的通用 LLM API 来使用（例如用 Postman 发请求让它写代码）。
* **用户的视角与反驳**："我们可以把各种 prompt 都放在服务器端，这样即使被盗用，用户得到的也是不稳定的输出，很不值得。"
* **最终技术决策**：**Server-side System Prompt Injection**。我们不会构建繁琐的签名鉴权。代理路由在接收到前端发来的 messages 时，将强行在后端拼装或覆盖具有极强"跑团 DM 属性"的 System Prompt。一旦被盗用，输出的内容将完全被跑团语境污染，彻底丧失作为"通用生产力工具"的价值，从源头消灭薅羊毛的动机。

#### 2. "物理开挂"与作弊防御 (State Trust & Cheating)
* **引发讨论的挑战**：因为 `roll_dice` 等工具在前端运行，懂技术的用户可以拦截网络请求，强行将判定结果修改为"大成功"。客户端的数据是"不可信"的。
* **用户的视角与反驳**："我不担心这个，比我们值得开挂的地方多了去了，也是个不经济的事情。"
* **最终技术决策**：**主动忽略防作弊机制**。本作定位为注重个人体验与桌游乐趣的应用，作弊只会破坏玩家自己的沉浸感。我们不为此投入任何不必要的"防篡改、数据校验"开发成本，完全拥抱甚至允许"本地魔改"。

#### 3. 跑团长记忆与 Context Window 爆炸 (Long-term Memory)
* **引发讨论的挑战**：跑团动辄几万 Token，如果在后端我们可以做 RAG（向量搜索）或自动 Summarization。而在前端做这些，意味着沉重的本地计算和巨大的 JS 内存开销。
* **用户的视角与反驳**："我计划用'隐藏的' buff/debuff 来做。比如玩家解救了公主，他就会得到 buff '与魔王战斗时国王80%会派人来帮忙'，用这样的简化手法替代搜索。"
* **最终技术决策**：**RPG 状态标签化机制（Tagging）**。这是极其巧妙的游戏化降维方案。我们将赋予 DM Agent 一个颁发 `buff/debuff` 的内部工具。当关键事件发生时，直接将庞大剧情提炼为一条属性附着在玩家身上。大模型在后续判别时，只需读取极为简练的 Buff 列表，彻底抛弃笨重的 RAG 架构。

#### 4. 第三方 API Key 代理导致接口增殖 (Multi-Service Proxy)
* **引发讨论的挑战**：为了生成地图或立绘，必须调用 Midjourney 等外部接口，由于 API Key 不能下发，我们必须在后端写大量专门的 Proxy Route 来包装这些独立技能。
* **用户的视角与反驳**："我倒觉得不是坏事。外部技能函数化反而使得更容易测试。同时，技能能够纯前端最好，必要的才结合服务器端。"
* **最终技术决策**：**前端优先，必要时微服务化**。我们的基本原则是：**所有技能（Tools）默认在纯前端实现**（如掷骰子、建 NPC、挂 Buff）。只有当技能必须隐藏敏感信息（如调用外部付费画图 API 时），才将其剥离为独立的 Next.js 后端 API Route。这种"前端主导 + 后端辅助"的拆分，既保持了最高效的 UI 联动，又让那些复杂的外部技能变得极易单独 Mock 和进行单元测试。

## 架构修正：Agent 运行环境从 Browser 迁至 Edge Runtime（2025-04）

### 背景
在实际引入 `@openai/agents` 后发现，该 SDK 底层深度依赖 `openai` Node.js SDK 的完整运行时（包括追踪 Tracing、Span 上下文管理等模块），其 import 链路中可能静态引用 `node:fs` 等原生模块。在 Next.js 的 `'use client'` 组件中引入，webpack 会尝试解析所有静态 import 并报错——**即使我们的代码从未调用那些路径。**

### 讨论路径
- **方案 A - 纯浏览器**：需要 webpack polyfill hack，构建期风险高，SDK 官方不保证
- **方案 B - Edge Runtime API Route**：`@openai/agents` 官方明确支持 Edge Runtime（`package.json` 有条件导出），不会拉入 `node:*` 模块。API Key 天然在服务端

### 最终决策
**Agent Runner 运行在 Edge Runtime API Route 中**（`/api/chat/route.ts`）。理由：
1. 构建安全：Edge 打包器走 SDK 的 Edge 导出路径
2. 密钥安全：不再需要独立 Proxy 层
3. 部署兼容：项目使用 `@opennextjs/cloudflare`（Edge 环境）
4. 工具安全：`roll_dice` 等工具是纯数学计算，零 Node 依赖

### 影响
- 废弃：`/api/llm-proxy/route.ts`（独立代理层不再需要）
- 废弃：`src/lib/ai/use-dm-agent.ts`（前端 Agent Hook 方案）
- 新增：SSE 事件流中增加 `tool_call` / `tool_result` 事件类型，前端通过这些事件渲染 ToolCard UI
- **前端的角色**从"Agent 执行者"变为"Agent 事件消费者"——接收 SSE 流，展示文本和工具交互卡片
